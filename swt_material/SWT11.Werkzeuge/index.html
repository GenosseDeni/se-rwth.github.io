Softwaretechnik, 11. Werkzeuge

Material für die Softwaretechnik Vorlesung, Bernhard Rumpe

Werkzeuge für die Softwareentwicklung

  • Software ist ein Werkstoff
  • Zu ihrer Bearbeitung benötigt man Werkzeuge

Git zeichnet sich besonders durch seine dezentrale Natur aus:

  • Jeder besitzt eine vollständige Kopie des Repositories
  • Gemeinsame Verwaltung aber auch lokale Revision des Repositories

Das “Remote” Repository ist dabei auch nur ein ganz normales Repository. Wir schauen uns diesen Umstand im Selbstlernkurs genauer an.

Neu angelegte Dateien werden nicht von Git verwaltet, nicht “getracked”. Erst nach explizitem hinzufügen, “add”, stehen sie unter der Kontrolle von Git. Wird eine Datei verändert, gilt sie als modifiziert. Erst wenn diese Änderungen sozusagen in den Mittelpunkt gerückt werden, “gestaged werden”, werden sie in der nächsten Version berücksichtigt.

Im Folgenden werden die zentralen Git-Kommandos übersichtlich darfgestellt.

Eine etwas ausführlichere Übersicht und eine reale Nachstellung der Befehle finden Sie auch im Selbstlernkurs.

Eine gemeinsame Verwaltung führt früher oder später zu Konflikten. Mechanismen zu deren Behebung finden Sie im Selbstlernkurs.

  • Eintönige, sich wiederholende Arbeit durch Automatisierung ersetzen
  • Fehler vermeiden, Abhängigkeiten berücksichtigen
  • Automatisierung ist essentiell für effiziente Entwicklung!

Make liefert inkrementelle Kochrezepte. Eine kurze Übersicht der wesentlichen Bestandteile eines Make-Scripts finden Sie im Selbstlernkurs.

Eine der Schwachstellen von Make ist das fehlende Dependency-Management. Hier hat Apache Ant und sein spiritueller Nachfolger Apacha Maven Abhilfe geschaffen:

  • Dargestellte Sequenz ist aus Darstellungsgründen nicht vollständig!
  • Unterschied zwischen “install” und “deploy”: “install” kopiert das gebaute Artefakt ins das lokale Maven Repository, “deploy” kopiert es in ein spezifiziertes Repository (bspw. Gitlab Package Registry)
  • Maven organisiert Abhängigkeiten zwischen Projekten
    • auch für importierte Repositories (also in Richtung anderer Projekte)
  • Dependency Management
    • zwischen Repositories (pro Maven-Projekt eines)
  • Snapshot-Verwaltung
    • Snapshot: noch nicht released aber Kandidat
  • In öffentlichen Repositories (Maven Central, Nexus) sind annähernd alle bekannten Java Libraries enthalten
    • Maven Central ist in jeder Maven POM standardmäßig aktiv
    • Plugins sind ebenfalls meistens auch auf Maven Central zu finden
      • Unternehmen haben lokale Repositories

Eine wesentliche Schwäche von Maven ist seine (fehlende) Flexibilität und Fähigkeit des inkrementellen Bauens. Im folgenden betrachten wir deswegen das Tool Gradle.

Automatisierung benötigt über das Build- und Dependency-Management hinaus Werkzeuge, die regelmäßig eine Integration und Bereitstellung (Deployment) ausführen. Diese bieten…

  • Build-Stabilität
  • Modul-Integration
  • Automatisiertes Testen -> kurze Test-Zyklen
  • Qualitäts-Analysen: Code-Smells, Commit-Historie
  • Reporting – Benachrichtigung im Fehlerfall
  • Automatisierte Verteilung auf Testsysteme

Gitlab ist ein webbasiertes Projektmanagement-Werkzeug. Es integriert viele Dienste:

  • Git als Versionskontrolle/Repository
  • Wiki
  • Bugreporting
  • Repository-Browser
  • Timeline / Roadmap / Meilensteinplanung
  • Automatisierung
    • Continuous Integration, CI pipelines
    • Generieren, compilieren, testen, prüfen, …

Probieren Sie Gitlab selbst kostenlos aus unter https://git.rwth-aachen.de/
